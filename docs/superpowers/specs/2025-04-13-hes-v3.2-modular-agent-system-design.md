# HES v3.2 — Modular Agent System Design

> Design Document — Modular multi-agent architecture with IDE auto-detection and session lifecycle management.
> Date: 2025-04-13 | Author: Josemalyson Oliveira | 2026

---

## ◈ PROBLEM STATEMENT

HES v3.1 uses a single-agent model where the LLM in session performs all tasks: orchestration, discovery, implementation, testing, and review. This creates several issues:

1. **Context bloat**: SKILL.md grows unbounded as features are added
2. **No parallelism**: Multiple features or sub-tasks cannot run concurrently
3. **No agent extensibility**: Users cannot add specialized agents (e.g., graphify for dependency analysis)
4. **Manual IDE setup**: Installation requires copying files and configuring each IDE by hand
5. **Session corruption risk**: No safe mechanism to `/clear` or `/new` and resume without losing state or re-doing work

---

## ◈ SOLUTION OVERVIEW

Introduce a **modular agent layer** that transforms HES from a single-agent to a multi-agent system while keeping it skill-file based (no CLI dependency).

**Four pillars:**

| Pillar | What | Why |
|--------|------|-----|
| **Agent Registry** | `.hes/agents/registry.json` — defines all agents, capabilities, triggers | Single source of truth for dispatching |
| **Phase Agents** | Specialized agents per HES phase (discovery, spec, design, test, impl, review) | Focused context, less bloat per agent |
| **Session Manager** | `skills/session-manager.md` — lifecycle, `/new`, `/clear`, recovery, context bloat detection | Safe session hygiene and phase isolation |
| **IDE Auto-Detection** | Bootstrap detects IDE and generates correct config (`.cursorrules`, `CLAUDE.md`, `AGENTS.md`, etc.) | One-step installation, zero manual config |

## ◈ ARCHITECTURE

### Layer Model

```
┌─────────────────────────────────────────────────────────┐
│                    SKILL.md (Orchestrator)               │
│  → Reads state, routes to correct agent                 │
│  → Manages session lifecycle via Session Manager        │
│  → Enforces phase boundaries                            │
│  → NEVER implements — only dispatches, validates, advances │
└────────────┬────────────────────────────────────────────┘
             │ dispatch
             ▼
┌─────────────────────────────────────────────────────────┐
│                  AGENT REGISTRY                         │
│  .hes/agents/registry.json                              │
│  → Agent definitions + capabilities + triggers          │
│  → Built-in: harness-agent, phase-agents, sub-agents    │
│  → Extensible: custom agents (graphify, etc.)           │
└────────────┬────────────────────────────────────────────┘
             │
    ┌────────┼─────────┬──────────┐
    ▼        ▼         ▼          ▼
┌────────┐┌──────┐┌────────┐┌──────────┐
│Phase   ││Phase ││Phase   ││ Session  │
│Agent   ││Agent ││Agent   ││ Manager  │
│01-04   ││05-06 ││07      ││          │
└────────┘└──────┘└────────┘└──────────┘
```

### Key Boundaries

| Component | Responsibility | Does NOT |
|-----------|---------------|----------|
| **Orchestrator** (SKILL.md) | Knows state, routes, delegates | Implements code, writes tests |
| **Agent Registry** (.hes/agents/) | Defines capabilities, triggers | Contains behavior logic |
| **Phase Agents** (skills/) | Execute specific phases with focused context | Route to other phases |
| **Session Manager** | Controls `/new`, `/clear`, recovery, context bloat | Implements features |
| **Sub-Agents** (test-runner, linter, arch-check) | Handle narrow tasks within a phase | Operate independently of parent |

## ◈ COMPONENTS

### 1. Agent Registry (`.hes/agents/registry.json`)

Central definition of all agents the harness can dispatch to.

```json
{
  "version": "3.2.0",
  "agents": {
    "harness-agent": {
      "description": "Default orchestrator agent — reads SKILL.md, routes phases",
      "type": "orchestrator",
      "triggers": ["/hes", "/hes status", "/hes start"],
      "context_load": ["SKILL.md", ".hes/state/current.json"]
    },
    "discovery-agent": {
      "description": "Runs 01-discovery — captures RN, use cases, domain analysis",
      "type": "phase",
      "phase": "DISCOVERY",
      "triggers": ["feature state = DISCOVERY"],
      "context_load": ["SKILL.md", "skills/01-discovery.md", ".hes/domains/*/context.md"]
    },
    "spec-agent": {
      "description": "Runs 02-spec — BDD scenarios, API contracts, RN traceability",
      "type": "phase",
      "phase": "SPEC",
      "triggers": ["feature state = SPEC"],
      "context_load": ["skills/02-spec.md"]
    },
    "design-agent": {
      "description": "Runs 03-design — components, ADRs, architecture decisions",
      "type": "phase",
      "phase": "DESIGN",
      "triggers": ["feature state = DESIGN"],
      "context_load": ["skills/03-design.md", ".hes/decisions/"]
    },
    "test-agent": {
      "description": "Runs 05-tests — RED phase, writes failing tests",
      "type": "phase",
      "phase": "RED",
      "triggers": ["feature state = RED"],
      "context_load": ["skills/05-tests.md", "skills/02-spec.md"]
    },
    "impl-agent": {
      "description": "Runs 06-implementation — GREEN phase, writes production code",
      "type": "phase",
      "phase": "GREEN",
      "triggers": ["feature state = GREEN"],
      "context_load": ["skills/06-implementation.md", ".hes/specs/*/02-spec.md"],
      "sub_agents": ["test-runner", "linter", "arch-check"]
    },
    "review-agent": {
      "description": "Runs 07-review — 5 dimensions, DONE gate",
      "type": "phase",
      "phase": "REVIEW",
      "triggers": ["feature state = REVIEW"],
      "context_load": ["skills/07-review.md", "skills/05-tests.md", "skills/06-implementation.md"]
    },
    "session-manager": {
      "description": "Manages session lifecycle — /new, /clear, context bloat detection, recovery",
      "type": "system",
      "triggers": ["/hes status", "context > threshold", "phase lock violation"],
      "context_load": ["skills/session-manager.md"]
    }
  },
  "sub_agents": {
    "test-runner": {
      "description": "Executes tests, reports results, loops on failure (max 3)",
      "parent": "impl-agent",
      "trigger": "after code change",
      "scope": ["tests/", "build output"]
    },
    "linter": {
      "description": "Runs linter + complexity check",
      "parent": "impl-agent",
      "trigger": "before commit",
      "scope": ["*.java", "*.ts", "*.py"]
    },
    "arch-check": {
      "description": "Validates architecture boundaries (ArchUnit/dep-cruiser/import-linter)",
      "parent": "impl-agent",
      "trigger": "before commit",
      "scope": ["architecture rules"]
    }
  },
  "custom_agents": {}
}
```

**Custom agents** allow users to add their own agents (e.g., graphify for dependency graph analysis). They register in the `custom_agents` slot and can be triggered by the orchestrator.

### 2. Delegation Protocol (SKILL.md Routing)

The orchestrator follows a strict dispatch pattern:

```
PASSO 1 — LER ESTADO → .hes/state/current.json
PASSO 2 — CONSULTAR REGISTRO → .hes/agents/registry.json
PASSO 3 — IDENTIFICAR AGENTE → matching agent for current phase
PASSO 4 — CARREGAR CONTEXT → only what that agent needs
PASSO 5 — DELEGAR → run agent's skill-file
PASSO 6 — VALIDAR → check phase DONE criteria
PASSO 7 — AVANÇAR → log event → next agent
```

**Key rule:** The orchestrator NEVER implements. It ONLY dispatches, validates, and advances state.

### 3. Multi-Feature Parallel Support

The dependency graph in `current.json` enables parallel feature work:

```json
{
  "active_feature": "payment",
  "features": {
    "payment": "GREEN",
    "auth": "DONE",
    "billing": "SPEC"
  },
  "dependency_graph": {
    "billing": ["payment"]
  }
}
```

**Rules:**
- `/hes switch <feature>` changes focus without losing state
- Blocked features (dependencies not DONE) show warning
- Different features can have different active agents simultaneously (via `/hes switch`)

## ◈ SESSION MANAGER

### Session Lifecycle States

```
┌──────────────┐
│   ACTIVE     │ ← Normal session running
└──────┬───────┘
       │
       ├─ Context bloat detected (> threshold)
       │     ▼
       │  ⚠️ WARNING: Context approaching limit
       │     → Suggest: /hes checkpoint + /clear
       │
       ├─ Phase lock violation (agent tries to skip ahead)
       │     ▼
       │  🚫 BLOCKED: Cannot advance past current phase
       │     → "Complete phase requirements first, or /hes unlock --force"
       │
       ├─ /clear or /new
       │     ▼
       │  📝 Save checkpoint → .hes/state/session-checkpoint.json
       │     → "Cleared. Run /hes status to resume from checkpoint."
       │
       └─ /hes status (recovery)
             ▼
          Read checkpoint → Restore exact state
             → "Session restored. Feature: X | Phase: Y | Last action: Z"
```

### Checkpoint Schema (`.hes/state/session-checkpoint.json`)

```json
{
  "timestamp": "2025-01-01T12:00:00Z",
  "feature": "payment",
  "phase": "GREEN",
  "agent": "impl-agent",
  "last_action": "Generated PaymentService.java — awaiting test run",
  "completed_steps": ["02-spec approved", "03-design approved", "05-tests written (RED)"],
  "pending_steps": ["run tests", "implement service", "verify GREEN"],
  "context_summary": "Implementing payment service with PIX support. RN-01: must support Brazilian real.",
  "artifacts_created": ["src/PaymentService.java", "tests/PaymentServiceTest.java"],
  "context_tokens_remaining": 4000
}
```

### Phase Lock Mechanism

Each phase has an explicit gate preventing premature advancement:

| Transition | Gate | Violation Handling |
|------------|------|-------------------|
| DISCOVERY → SPEC | RN list approved by user | "BLOCKED: Complete RN discovery first" |
| SPEC → DESIGN | BDD scenarios + API contract approved | "BLOCKED: Write and approve spec first" |
| DESIGN → DATA | ADRs approved | "BLOCKED: Finalize design decisions" |
| DATA → RED | Migrations reviewed | "BLOCKED: Review migration safety" |
| RED → GREEN | ≥1 test failing (proof of RED) | "BLOCKED: Tests must fail first" |
| GREEN → REVIEW | Build + all tests passing | "BLOCKED: Fix build failures" |
| REVIEW → DONE | 5-dimension checklist complete | "BLOCKED: Complete review checklist" |

**Violation handling:** Agent tries to skip → BLOCKED with clear message → Options: [A] complete requirements [B] `/hes unlock --force` (logs risk event)

### Recovery Protocol

After `/clear` or `/new`:

1. Read `.hes/state/session-checkpoint.json`
2. Read `.hes/state/current.json`
3. Compare consistency:
   - **YES** → "Checkpoint valid. Resuming from {{phase}} of {{feature}}"
   - **NO** → "⚠ Checkpoint inconsistency. Using current.json as source of truth."
4. Load ONLY the skill-file for current phase
5. Show pending steps
6. "Ready to continue. Last action: {{last_action}}"

### Context Bloat Detection

Heuristics that trigger a `/clear` suggestion:

| Heuristic | Threshold | Action |
|-----------|-----------|--------|
| Messages in session | > 50 | Suggest checkpoint + clear |
| User says "start fresh" / "too much context" | 1 match | Suggest checkpoint + clear |
| Agent re-reads same file | 3+ times | Suggest checkpoint + clear |
| Skill-file references earlier messages | "as we discussed before..." | Suggest checkpoint + clear |

When triggered:
```
⚠ Context is getting heavy ({{N}} messages). Suggest:
  1. I'll save a checkpoint with current progress
  2. You run /clear
  3. I'll resume from checkpoint with clean context

  [A] Save checkpoint + clear  [B] Continue for now
```

## ◈ IDE AUTO-DETECTION & BOOTSTRAP

### Detection Flow

Bootstrap (`00-bootstrap.md`) detects IDE in Passo 1.5:

```
Detection order:
1. Check for marker files/dirs:
   - .vscode/ → VS Code
   - .cursor/ or .cursorrules → Cursor
   - .claude/ or CLAUDE.md → Claude Code
   - .gemini/ → Gemini CLI
   - .openhands/ → OpenHands
   - .windsurf/ → Windsurf

2. If multiple detected → ask user:
   "Detected: VS Code + Cursor. Which is your primary IDE?"

3. If none detected → ask user:
   "What IDE/editor are you using?"
   [A] Claude Code  [B] Cursor  [C] VS Code  [D] Windsurf  [E] Copilot  [F] Other
```

### IDE-Specific Config Generation

| IDE | Config File | Content |
|-----|-------------|---------|
| **Claude Code** | `.claude/CLAUDE.md` | Read SKILL.md → current.json → dispatch per registry → enforce phase locks |
| **Cursor** | `.cursorrules` | Same instructions as CLAUDE.md but in Cursor format |
| **VS Code** | `.vscode/hes-agent.md` + `.vscode/settings.json` | Instructs Copilot/agent extension |
| **Windsurf** | `.windsurfrules` | Same pattern as .cursorrules |
| **Generic** | `AGENTS.md` | Generic agent instruction file — read SKILL.md first, follow routing |

### Bootstrap Enhancement — New Passos

| Passo | Action |
|-------|--------|
| **1.5** | IDE Auto-Detection → detect environment, generate correct config file(s), register in current.json: `"ide": "claude-code\|cursor\|vscode\|..."` |
| **1.6** | Agent Registry Init → generate `.hes/agents/registry.json`, register default agents, empty custom_agents slot |
| **1.7** | Session Manager Init → generate `skills/session-manager.md`, generate empty `.hes/state/session-checkpoint.json`, register session-manager in registry |

### Updated `current.json` Schema

## ◈ ERROR HANDLING

### Phase Lock Violation

```
🚫 PHASE LOCK VIOLATION

You attempted to {{ACTION}} but the current phase is {{PHASE}}.
Required to advance: {{LIST_OF_MISSING_REQUIREMENTS}}

Options:
  [A] Complete the missing requirements (recommended)
  [B] /hes unlock --force (bypass — logs risk event, NOT recommended)

Risk: Advancing without completing requirements may cause rework or missed edge cases.
```

### Checkpoint Inconsistency

```
⚠ CHECKPOINT INCONSISTENCY DETECTED

Checkpoint says: Feature={{X}}, Phase={{Y}}
Current state says: Feature={{X}}, Phase={{Z}}

Using current.json as source of truth.
Last consistent action: {{LAST_ACTION_FROM_EVENTS_LOG}}

[ A] Continue from current.json state
[ B] Restore from checkpoint (may lose recent work)
```

### Agent Not Found

```
⚠ AGENT NOT FOUND

Phase: {{PHASE}}
Expected agent: {{AGENT_NAME}}
Status: Not found in registry

[ A] Use default agent (harness-agent)
[ B] Register agent in .hes/agents/registry.json
```

## ◈ CUSTOM AGENTS

Users can extend the system with custom agents. The `custom_agents` slot in the registry accepts user-defined agents:

```json
{
  "custom_agents": {
    "graphify-agent": {
      "description": "Analyzes dependency graphs, suggests module boundaries",
      "reference": "https://github.com/safishamsi/graphify",
      "type": "custom",
      "triggers": ["/hes analyze", "design-agent requests analysis"],
      "context_load": ["skills/custom/graphify-agent.md"],
      "output": ".hes/domains/{{domain}}/analysis.md"
    }
  }
}
```

**Creation flow:**
1. Add entry to `custom_agents` in registry
2. Create `skills/custom/<agent-name>.md` with behavior definition
3. Trigger via `/hes run <agent-name>` or automatic dispatch from parent agent

## ◈ CONSTRAINTS & LIMITATIONS

| Constraint | Impact |
|-----------|--------|
| **Skill-file based only** | No CLI or external dependencies required. Everything runs through LLM context. |
| **Single LLM session** | "Multi-agent" means role delegation within one session. True parallel execution requires `/hes switch` between features. |
| **Context window** | Each agent loads only its required context. Session manager handles bloat. |
| **No real-time coordination** | Sub-agents run sequentially within a parent agent's context. No concurrent execution. |

---

## ◈ SUCCESS CRITERIA

| Metric | Target |
|--------|--------|
| SKILL.md line count | Reduced from current ~500 to ≤ 300 (routing only) |
| IDE setup time | From ~5 minutes manual → < 1 minute auto-detect |
| Session safety | `/clear` + `/hes status` recovers exact state without rework |
| Phase isolation | Agent cannot skip phases — 100% lock enforcement |
| Agent extensibility | Custom agents can be added without modifying core skill-files |
