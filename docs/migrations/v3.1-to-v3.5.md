# Migration Guide: HES v3.1 → v3.5

> If you're running HES via a skill installed at `~/.claude/skills/hes/` or  
> `/mnt/skills/user/harness-engineer/SKILL.md`, you are likely on v3.1.  
> This guide covers everything that changed and how to update.

---

## Breaking Changes

### 1. SECURITY phase added to state machine

**v3.1 state machine:**
```
ZERO → DISCOVERY → SPEC → DESIGN → DATA → RED → GREEN → REVIEW → DONE
```

**v3.5 state machine:**
```
ZERO → DISCOVERY → SPEC → DESIGN → DATA → RED → GREEN → SECURITY → REVIEW → DONE
```

**Impact**: Features in GREEN that skip SECURITY and go directly to REVIEW are **no longer valid** in v3.5.  
`current.json` features with state `REVIEW` that were previously at `GREEN` may have skipped the gate.

**Fix**: Any active feature at `GREEN` must pass through `skills/10-security.md` before advancing.

---

### 2. ORPHAN state (new)

v3.1 had no ORPHAN state — a missing `current.json` with `.hes/` present would cause undefined behavior.

v3.5 defines explicitly:
```
ORPHAN → .hes/ exists, no current.json → load skills/legacy.md
```

**Impact**: First session after clone now reliably lands on `legacy.md` instead of undefined behavior.

---

### 3. LLM Execution Mandate (new in v3.5)

v3.1: "Read this file, route to skill-file, then execute."  
v3.5: "**You ARE the harness.** Read this file IN FULL. YOU execute every step autonomously."

The key shift: v3.5 explicitly forbids asking the user to run steps. The LLM must perform file reads, writes, git operations, and state transitions using its tools directly.

---

### 4. `current.json` schema expanded

v3.1 minimal template:
```json
{"phase":"ZERO","active_feature":null,"features":{},"harness_version":"3.1.0"}
```

v3.5 full schema adds:
- `user_language` — auto-detected, adapts all responses
- `audience_mode` — `"expert"` or `"guided"` 
- `session.phase_lock` — prevents re-running a phase before gate passes
- `session.messages_in_session` — step budget tracker
- `security.last_scan` / `last_gate_result` — SECURITY phase tracking
- `step_budget` — per-phase max steps (DISCOVERY=15, SPEC=20, etc.)
- `token_tracking` — cost estimation
- `model`, `stack`, `ide` — runtime context

**Impact**: v3.1 `current.json` files are readable by v3.5 (backwards compatible) but missing fields cause undefined behavior when the LLM accesses budgets.

---

### 5. Agent Registry (new in v3.5)

v3.5 routes to named agents via `.hes/agents/registry.json`.  
v3.1 routed directly to skill-files.  
**Impact**: None if SKILL.md is being used — the registry lookup is internal.

---

### 6. Step Budget Protocol (new in v3.5)

v3.5 enforces per-phase step limits. After `max` steps, the LLM must:
1. Declare budget exhausted
2. Write a summary of what was accomplished
3. Ask the user to confirm continuation or adjust scope

v3.1 had no step limits.

---

## How to Update

### Option A — Reinstall via setup (recommended)

```bash
cd /tmp/hes && git pull
./setup --tools claude --scope global
```

This overwrites `~/.claude/skills/hes/SKILL.md` with v3.5.

### Option B — Manual copy

```bash
cp /path/to/hes/.agents/skills/hes/SKILL.md ~/.claude/skills/hes/SKILL.md
```

### Option C — Claude.ai / App (no filesystem access)

In **Settings → Project → Instructions**, replace your current instructions with:

```
You are a HES Harness Engineer (v3.5.0). When /hes is invoked:
1. Read SKILL.md in full — you are the harness, not a router
2. Check .hes/state/current.json — handle ORPHAN state if file is missing
3. Load skills/<phase>.md and execute autonomously via your tools
4. NEVER ask the user to perform steps — YOU execute them
5. State machine: ZERO → DISCOVERY → SPEC → DESIGN → DATA → RED → GREEN → SECURITY → REVIEW → DONE
```

## Migrating existing `current.json` to v3.5 schema

If you have a project with an active feature at v3.1 schema, add the missing fields:

```bash
# In your project root
python3 - << 'EOF'
import json, datetime

with open(".hes/state/current.json") as f:
    state = json.load(f)

# Add missing v3.5 fields (only if not present)
state.setdefault("user_language", None)
state.setdefault("audience_mode", "expert")
state.setdefault("session", {"checkpoint": None, "phase_lock": None, "messages_in_session": 0})
state.setdefault("security", {"last_scan": None, "last_gate_result": None, "exceptions_count": 0})
state.setdefault("step_budget", {
    "DISCOVERY": {"max": 15, "used": 0},
    "SPEC":      {"max": 20, "used": 0},
    "DESIGN":    {"max": 20, "used": 0},
    "DATA":      {"max": 15, "used": 0},
    "RED":       {"max": 25, "used": 0},
    "GREEN":     {"max": 30, "used": 0},
    "SECURITY":  {"max": 10, "used": 0},
    "REVIEW":    {"max": 15, "used": 0}
})
state.setdefault("token_tracking", {"tokens_estimated": 0, "cost_usd_estimated": 0.0})
state.setdefault("model", None)
state.setdefault("stack", None)
state.setdefault("ide", None)
state.setdefault("domains", [])
state.setdefault("dependency_graph", {})
state["harness_version"] = "3.5.0"
state["last_updated"] = datetime.datetime.utcnow().isoformat() + "Z"

# If active feature is at GREEN and hasn't gone through SECURITY, flag it
if state.get("active_feature"):
    feat = state["active_feature"]
    if state.get("features", {}).get(feat) == "REVIEW":
        print(f"WARNING: Feature '{feat}' is at REVIEW but may have skipped SECURITY gate.")
        print("Recommend reviewing security posture before proceeding.")

with open(".hes/state/current.json", "w") as f:
    json.dump(state, f, indent=2)

print("Migration complete: current.json updated to v3.5 schema")
EOF
```
